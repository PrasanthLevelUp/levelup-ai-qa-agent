# Add Employee — Gold Standard

The **single-requirement gold standard** for Phase 1. This is the bar: *a senior QA
architect cannot easily think of an important scenario that LevelUp missed.*

Not a leaderboard. Not a score to optimise. It is the exhaustive list of scenarios
an experienced QA architect would expect a test suite for **Add Employee** to
contain, so we can generate, compare, and fix whatever is missing — one capability
at a time.

## The requirement under test

> **Add Employee** (HR / Employee Management)
> An authorized admin adds a new employee by entering **first name**, **last name**,
> an **employee ID**, and an **optional profile photo**. The employee ID may be
> **auto-generated or entered manually** and **must be unique**. On save the record
> is created, a **success notification** is shown, the admin is **redirected to the
> employee list**, and the new employee becomes **searchable by ID and by name**.

## How to read this file

- Every scenario a senior QA would expect is listed under its category.
- **Priority** — `Critical` (a defect here is a serious escape) → `High` → `Medium`
  → `Low` (polish / nice-to-have).
- **Detection keywords** — how the audit harness (`scripts/add-employee-audit.ts`)
  decides the generated suite covers this scenario. *Any one* phrase present in a
  generated scenario's title/objective/risk-area (token-stemmed, via the shared
  `coverage-match` matcher) counts as covered. These describe the concept — they
  are **not** a script to match verbatim.
- **Grounding** — `explicit` = the requirement states it directly; `expected` = a
  senior QA expects it for an HRM Add-Employee form even though the one-paragraph
  requirement does not spell it out. A miss on an `expected` scenario is a real
  conversation ("should the generator raise this?"), not automatically a defect.
